1. HttpClient (Angular HTTP API)
--------------------------------
Angular provides HttpClient service to communicate with backend API's.

import {HttpClient} from '@angular/common/http';

Why HttpClient?
It provides
* Typed responses
* RxJS Observables
* Interceptors support
* Automatic JSON parsing
* Error handling

Common HttpClient methods
Method      Purpose
-------------------
GET         fetch data
POST        Create data
PUT         Replace data
Patch       Update partial data
Delete      Remove data

Example GET Request
this.http.get<Product[]>('/api/products')

Breakdown:
this.http --> HttpClient instance
.get --> HTTP GET Request
<Product[]> --> Typescript generic
/api/products - API endpoint

Return type:
Observable<Product[]>

Meaning:
Response arrives asynchronously as a stream.

POST Example
------------
this.http.post<Product>('/api/products',product)

Meaning:
* Send product object
* Server returns created product

4. Retry operator
-----------------
Sometimes API fails temporarily

5 Caching concept
-----------------
Instead of calling API again:
Store response in memory

Example:
if(data cached)
   return cached Observable
   else
   call API

This improves performance

What is HttpClient?

HttpClient is Angular's service for performing HTTP requests. It returns Observables, supports interceptors, typed responses, and integrates with RxJS for reactive programming.

🎤 What is Interceptor?

An interceptor is middleware that intercepts HTTP requests and responses globally. It is used to modify requests, add authentication tokens, handle errors, implement caching, or log requests.

🎤 Why Clone Request?

HttpRequest objects are immutable, meaning they cannot be modified directly. Therefore we clone the request and apply modifications like adding headers.

1. Problem with Manual subscribe()
current code
-----------
detailProduct(){
    this.productService.getproductDetail().susbscribe({
        next:(data)=>{
            this.products = data;
        },
        error:(err)=>{
            console.error(err)

this works but has hidden Problem

Angular does not automatically unsubscribe
If we forget to unsubscribe:
memory leak

Example scenario:
Component destroyed
Observable still running
memory keeps increasing

2.Traditional Solution(Before Async Piple)
----------------------------------------
Developers had to manually unsubscribe

subscription!: subscription;

ngOnInit(){
    this.subscription = this.productService.getDetail().subscribe(data=>{
        this products =data;
    })
}
ngOnDestroy(){
    this.subscription.unsubscribe();
}
this becomes messy in large apps

3. Async Pipe Solution
----------------------
Instead of subscribing in Typescript, we let Angular subscribe automatically.

product$!:Observable<Product[]>;

ngOnInit(){
    this.products$ = this.productService.getDetails()
}

Notice: product$
the $ symbol indicates Observable stream

HTML
<ul>
<li *ngFor="let p of product$ |async">
    {{p.name}}
 </li>
</ul>

7️⃣ Angular Interview Answer

If interviewer asks:
Why use async pipe instead of subscribe?
Answer:
The async pipe automatically subscribes and unsubscribes from observables in Angular templates. This prevents memory leaks and keeps the component code cleaner.

1️⃣ switchMap (Most Used in Angular)
Behavior

When a new value comes → cancel previous request.

Example scenario: search box / route param changes

Example
this.route.paramMap
  .pipe(
    switchMap(params =>
      this.productService.getProductDetail(params.get('id')!)
    )
  )
  .subscribe(product => {
    this.product = product;
  });
Flow
User clicks
/products/1
/products/2
/products/3

switchMap does:

API(1) ❌ cancelled
API(2) ❌ cancelled
API(3) ✅ executed
When to Use

✔ Search suggestions
✔ Route params
✔ Live filtering
✔ Typeahead

2️⃣ mergeMap
Behavior

All requests run in parallel.

Example: loading multiple products.

Example
from(productIds)
  .pipe(
    mergeMap(id => this.productService.getProductDetail(id))
  )
  .subscribe(product => {
    console.log(product);
  });

If IDs are:

[1,2,3]

Requests:

API(1)
API(2)
API(3)

All run simultaneously.

Result order may be:

3
1
2
When to Use

✔ Multiple API calls
✔ Batch requests
✔ Parallel operations

3️⃣ concatMap
Behavior

Requests run one after another in order.

Example: processing orders.

Example
from(orderIds)
  .pipe(
    concatMap(id => this.orderService.processOrder(id))
  )
  .subscribe(result => {
    console.log(result);
  });

Requests:

API(1) → finish
API(2) → start
API(3) → start

Order preserved.

Output:

1
2
3
When to Use

✔ Sequential operations
✔ Payment processing
✔ Order processing

4️⃣ Visual Comparison
Operator	Behavior	Cancels Previous	Order
switchMap	keeps latest	✔	latest only
mergeMap	parallel	❌	random
concatMap	queue	❌	preserved

6️⃣ Angular Interview Question

Q: Difference between switchMap and mergeMap?

Answer:

switchMap cancels the previous observable when a new value arrives, while mergeMap allows multiple observables to run in parallel without cancellation.

7️⃣ Easy Memory Trick
switchMap → switch to latest
mergeMap → merge all requests
concatMap → queue requests
