3. BehaviourSubject ***
-----------------------

RxJs provides special observable called:BehaviourSubject

It stores current value

Eg:
const subject = new BehaviorSubject(0);
Meaning:
initial state = 0;

Subscribers recieve current value immediately
Eg:
subject.subscribe(value=>console.log(value));
output: 0

if we update state:
subject.next(1)
output:1
All subscribers get Updated

BehaviorSubject vs Subject
--------------------------

| Subject                         | BehaviorSubject               |
| ------------------------------- | ----------------------------- |
| No initial value                | Requires initial value        |
| No last value                   | Stores last value             |
| Subscribers miss previous value | Subscribers get current value |

Q. What is state management?
A. State management is a pattern used to manage and share application data in a predictable and centralized way across multiple components.

Q. Why use BehaviorSubject?
A. BehaviorSubject stores the latest value and immediately emits it to new subscribers, making it ideal for maintaining application state.

Q. Why expose observable instead of subject?
A. Exposing only observable using asobservable() prevents external components from mutating the state directly.

Q. Explain store pattern
A. Store pattern centralizes application state in a single service. Components dispatch actions to update state and they subscribe to observables to react to state changes.

Q. Difference between store & service
A. Service:
logic
API calls

state container

------------------------------------
understanding Service vs Store clearly is very important for Angular architecture and interviews.

Many developers confuse them because a store is usually implemented using a service. The difference is how the service is used and what responsibility it has.

Let’s break it down step-by-step.

1️⃣ What is a Service in Angular
A service is simply a class used to share logic, utilities, or data access across components.
Angular provides Dependency Injection so that services can be injected anywhere.

Example service:
@Injectable({
  providedIn: 'root'
})
export class ProductService {

  constructor(private http: HttpClient) {}

  getProducts() {
    return this.http.get<Product[]>('/api/products');
  }
}

What this service does
Component
   ↓
Service
   ↓
API

Responsibilities of a typical service:
• API calls
• business logic
• data transformation
• helper functions

Example Component Using Service
export class ProductComponent {

  products: Product[] = [];

  constructor(private productService: ProductService) {}

  ngOnInit() {

    this.productService.getProducts()
      .subscribe(data => {
        this.products = data;
      });
  }
}

Flow:
Component requests data
↓
Service calls API
↓
Component stores response locally
Problem with Only Using Services

Imagine this app:

ProductListComponent
CartComponent
CheckoutComponent
HeaderComponent

All need product data.

If each component calls the service separately:
ProductList → API call
Cart → API call
Checkout → API call

Problems:
• Multiple API calls
• Data duplication
• State inconsistencies
• Hard to track updates

2️⃣ What is a Store?

A store is a centralized state container.
Instead of each component storing its own data:

Components
     ↓
Central Store

The store holds application state and exposes it as observables.
